= µhLS µhIR hardware IR specification
:toc:
:toclevels: 3
:sectnums:

.tldr:
[quote]
____
µhIR is the human-readable, line-oriented hardware refinement IR family used by µhLS between µIR and backend glue generation. A design moves through `seq`, `alloc`, `sched`, `bind`, and `fsm`, with stable operation identities preserved across the refinement stages. All artifacts use the `.uhir` extension and declare their stage in-file; `exg` is auxiliary collateral for allocation.
____

== Overview

This document specifies the *µhIR* textual intermediate representation family used by *µhLS* for hardware-side refinement after canonical compiler IR processing.

The common file extension for all µhIR artifacts is:

[source,text]
----
.uhir
----

The refinement stage is declared *inside* the file via the `stage` header field.

Typical filenames are:

[source,text]
----
fir.seq.uhir
fir.alloc.uhir
fir.sched.uhir
fir.bind.uhir
fir.fsm.uhir
----

The intended flow is:

[source,text]
----
µIR
  -> seq.uhir
  -> alloc.uhir
  -> sched.uhir
  -> bind.uhir
  -> fsm.uhir
  -> .uglir
  -> rtl
  -> Verilog
  -> Yosys / ASIC flow
----

The five standard refinement µhIR stages are:

* `seq`   — hierarchical sequencing graph representation built from sequencing graph units (SGUs)
* `alloc` — resource-type allocated design representation
* `sched` — scheduled design representation
* `bind`  — resource-bound design representation
* `fsm`   — controller-oriented FSMD representation after binding

One auxiliary µhIR stage is also defined for collateral exchange:

* `exg`   — executability graph collateral used by allocation

== Design goals

µhIR is designed to be:

* human-readable
* line-oriented
* easy to diff
* easy to parse in Python
* explicit about hierarchy, dependencies, scheduling, and binding
* stable across refinement stages

A central design rule is:

____
The same operation identities should be preserved across `seq`, `alloc`, `sched`, `bind`, and `fsm`.
____

This means that a node such as `v17` should denote the same conceptual operation across all refinement stages.

== Common structure

All µhIR files share a common outer structure.

A file contains:

* a `design` declaration
* a `stage` declaration
* optional global declarations
* one or more `region` blocks
* optional resource declarations in `bind` and `fsm` stages

Minimal example:

[source,text]
----
design fir
stage seq

region R0 kind=procedure {
  node v1 = add a, b : i32
}
----

== Common lexical conventions

=== Comments

Comments begin with `#` and continue to the end of the line.

[source,text]
----
# this is a comment
----

=== Identifiers

Identifiers are used for:

* design names
* region names
* node names
* resource names
* symbolic attributes

Examples:

[source,text]
----
fir
R0
v17
mul0
true_path
----

Recommended style:

* regions: `R0`, `R1`, ...
* nodes: `v1`, `v2`, ...
* resources: `mul0`, `add0`, `r_acc`, ...

=== Types

Scalar examples:

[source,text]
----
i1
i8
i16
i32
u8
u16
u32
----

Memory-like examples:

[source,text]
----
memref<i32>
memref<i32, 4>
memref<u32>
----

When the original µIR memory object has one statically known extent, µhIR should
preserve that shape as `memref<T, N>`.
This shape information is intended to survive through later HLS stages so wrapper
generation and later memory-oriented transforms can distinguish problem-sized
memories from unbounded external interfaces.

Later µhIR memory partition passes may refine one logical
`memref<T, N>` into partitioned/banked memory resources and regions while
preserving the original logical memory identity.

Examples:

----

Lists use brackets:

[source,text]
----
input=[r0, r1, r2]
----

Examples:

== Common top-level declarations

=== Design declaration

Every file must begin with exactly one design declaration.

Syntax:

[source,text]
----
design <design-name>
----

Example:

[source,text]
----
design fir
----

=== Stage declaration

Every file must declare exactly one stage.

Syntax:

[source,text]
----
stage <seq|alloc|sched|bind|fsm|exg>
----

Examples:

[source,text]
----
stage seq
stage alloc
stage sched
stage bind
stage fsm
stage exg
----

=== Global declarations

Optional global declarations may be used to document interface objects.

Syntax:

[source,text]
----
input  <name> : <type>
output <name> : <type>
const  <name> = <literal> : <type>
----

Example:

[source,text]
----
input  A : memref<i32>
input  B : memref<i32>
output C : memref<i32>
const  N = 16 : i32
----

== Regions

Regions are the primary structural unit of µhIR.

In the `seq` stage, each region denotes one sequencing graph unit (SGU).
An SGU is a directed acyclic graph with explicit hierarchy.
Every SGU is polar: it contains exactly one source `nop` node and exactly one sink `nop` node.

A region may represent:

* a procedure
* a loop body or loop controller SGU
* a branch arm
* an empty alternative SGU
* a subgraph used for hierarchical scheduling

Syntax:

[source,text]
----
region <region-id> kind=<kind> [parent=<region-id>] {
  ...
}
----

Examples:

[source,text]
----
region R0 kind=procedure {
  ...
}

region R1 kind=then parent=R0 {
  ...
}
----

Suggested region kinds:

* `procedure`
* `then`
* `else`
* `loop`
* `body`
* `empty`
* `pipeline`
* `subgraph`

== Nodes

Nodes represent operations or special control constructs.

In the `seq` stage, nodes fall into three classes:

* polar boundary nodes labeled `nop`
* datapath operation nodes such as `add`, `mul`, `lt`, `load`, `store`
* hierarchical control nodes such as `call`, `branch`, and `loop`

General syntax:

[source,text]
----
node <node-id> = <op> <args> [: <type>] class=<resource-class> delay=<int> [additional-attributes]
----

Examples:

[source,text]
----
node v1 = add a, b      : i32 class=add   delay=1
node v2 = load A[i]     : i32 class=memrd delay=1
node v3 = store C[i], x       class=memwr delay=1
node v4 = mul v1, v2    : i32 class=mul   delay=2
----

Notes:

* `store` usually has no result type
* `nop` is reserved for SGU source and sink nodes
* `call` references exactly one child SGU that is executed once
* `loop` references exactly one child SGU that is executed zero to many times
* `branch` references exactly two child SGUs, of which exactly one is executed once
* hierarchical nodes are pseudo vertices in the parent SGU; child SGUs conceptually return to that same vertex, and parent-level execution continues after it
* the datapath belonging to a callee, a looped code region, or a branch arm lives in the child SGU, not in the parent pseudo vertex
* the exact operation vocabulary is intentionally lightweight and parser-friendly

== Edges

Edges represent relationships between nodes and regions.

Supported edge kinds:

* `data`    — value/data dependency
* `control` — control dependency
* `order`   — serialization or sequencing constraint
* `mem`     — memory dependence
* `seq`     — region sequencing relation
* `exg`     — executability relation in `exg` collateral

General syntax:

[source,text]
----
edge <kind> <src> (->|--) <dst> [when=true|false]
----

Examples:

[source,text]
----
edge data v1 -> v4
edge control c0 -> br0
edge order v2 -> v5
edge rel v3 -- v4
edge mem v3 -> v7
edge seq br0 -> R1 when=true
edge seq br0 -> R2 when=false
----

For the `seq` stage, the literature baseline is:

* `data` edges connect operations inside a single SGU
* `seq` edges connect SGU boundaries and hierarchical nodes to child SGUs
* a hierarchical node is both the dispatch point into its child SGU(s) and the join point from which execution continues in the parent SGU
* child SGUs contain the datapath of the called procedure, looped region, or selected branch arm
* explicit CFG-like `control` edges should be minimized in favor of `call`, `branch`, and `loop` hierarchy

== Source mapping

Source mapping is optional but recommended for provenance tracking.

Syntax:

[source,text]
----
map <node-id> <- <source-id>
----

Example:

[source,text]
----
map v3 <- %t17
----

This may refer to:

* a µIR value
* a frontend source label
* another implementation-defined source origin

== Stage: `seq`

=== Purpose

The `seq` stage is the allocation input representation.

It captures:

* a hierarchy of sequencing graph units (SGUs)
* datapath operations within each SGU
* data dependencies within each SGU
* hierarchical execution structure through `call`, `branch`, and `loop`
* explicit SGU boundaries via source and sink `nop` nodes

The `seq` stage is the natural place to represent the SG literature model directly.
It should not merely mirror the canonical µIR control-flow graph.
Instead, it should restore a hierarchical HLS-oriented sequencing graph whose regions are SGUs.

=== Required elements

A valid `seq` file must contain:

* `design`
* `stage seq`
* at least one `region`
* zero or more `node`, `edge`, and `map` statements inside regions
* for every SGU region, exactly one source `nop` node and exactly one sink `nop` node

Additional structural rules for `seq`:

* every SGU region is acyclic
* `call` nodes reference exactly one child SGU
* `loop` nodes reference exactly one child SGU
* `branch` nodes reference exactly two child SGUs
* child SGUs of a hierarchical node return to that same parent-level pseudo vertex
* child SGUs contain the datapath of the hierarchical construct itself; later parent-level operations remain sequenced after the pseudo vertex
* operations that execute after a hierarchical construct are sequenced after the hierarchical node itself, not after the child region identifier
* if one branch alternative is structurally empty, it should still be represented explicitly as an empty `nop -> nop` SGU

=== Allowed attributes on nodes

Optional attributes:

* width/type information via `: <type>`
* implementation-specific annotations

The canonical `seq` emission carries only structural sequencing attributes such as `role`, `child`, `true_child`, and `false_child`.
A later µhIR graph-optimization step such as `uhls gopt -p infer_static` may annotate trivially statically bounded `loop` hierarchy nodes with `static_trip_count=<int>`.
For branch-oriented graph rewrites, `seq` may also carry:

* `incoming=[<pred0>, <pred1>, ...]` on `phi` nodes, preserving predecessor labels in operand order
* `pred=<predicate>` on nodes introduced by `uhls gopt -p predicate`
* branch provenance such as `header_label`, `true_label`, `false_label`, `true_input_label`, and `false_input_label`

After predication, a join-like dataflow merge may be expressed with:

[source,text]
----
node v19 = sel c, a_t, a_f : i32
----

Interpretation:

* `phi` is a control-flow merge
  it associates incoming values with predecessor labels via `incoming=[...]`
* `sel` is a dataflow merge
  it chooses between already available true/false values via an explicit condition operand

where the three operands are:

* branch condition
* value for the true arm
* value for the false arm

Allocation attributes such as `class`, `ii`, and `delay` are introduced by the `alloc` stage.

=== Example: flat SGU

[source,text]
----
design fir
stage seq

input  A : memref<i32>
input  B : memref<i32>
output C : memref<i32>

region R0 kind=procedure {
  node v0 = nop                 role=source
  node v1 = load A[i]           : i32
  node v2 = load B[i]           : i32
  node v3 = mul  v1, v2         : i32
  node v4 = add  acc, v3        : i32
  node v5 = store C[i], v4
  node v6 = nop                 role=sink

  edge data v1 -> v3
  edge data v2 -> v3
  edge data v3 -> v4
  edge data v4 -> v5

  edge seq v0 -> v1
  edge seq v0 -> v2
  edge seq v3 -> v4
  edge seq v5 -> v6

  map v1 <- %t1
  map v2 <- %t2
  map v3 <- %t3
  map v4 <- %t4
}
----

=== Example: call SGU hierarchy

[source,text]
----
design call_example
stage seq

region R0 kind=procedure {
  node v0  = nop                 role=source
  region_ref R1
  node v1  = call cos_kernel          child=R1
  node v2  = mul x, gain         : i32
  node v3  = nop                 role=sink

  edge seq v0 -> v1
  edge seq v1 -> v2
  edge seq v2 -> v3
}

region R1 kind=subgraph parent=R0 {
  node v4 = nop                 role=source
  node v5 = mul a, b            : i32
  node v6 = sub v5, c           : i32
  node v7 = nop                 role=sink

  edge data v5 -> v6
  edge seq  v4 -> v5
  edge seq  v6 -> v7
}
----

=== Example: branch SGU hierarchy

[source,text]
----
design branch_example
stage seq

region R0 kind=procedure {
  node v0  = nop                 role=source
  region_ref R1
  region_ref R2
  node v1  = lt x, y             : i1
  node v2  = branch v1                true_child=R1 false_child=R2
  node v3  = mul z, 2            : i32
  node v4  = nop                 role=sink

  edge seq  v0 -> v1
  edge data v1 -> v2
  edge seq  v2 -> R1 when=true
  edge seq  v2 -> R2 when=false
  edge seq  v2 -> v3
  edge seq  v3 -> v4
}

region R1 kind=then parent=R0 {
  node v5 = nop        role=source
  node v6 = add a, b : i32
  node v7 = nop        role=sink

  edge seq v5 -> v6
  edge seq v6 -> v7
}

region R2 kind=else parent=R0 {
  node v8 = nop        role=source
  node v9 = sub a, b : i32
  node v10 = nop        role=sink

  edge seq v8 -> v9
  edge seq v9 -> v10
}
----

=== Example: loop SGU with explicit empty alternative

[source,text]
----
design loop_example
stage seq

region R0 kind=procedure {
  node v0  = nop                 role=source
  region_ref R1
  node v1  = loop                     child=R1
  node v2  = store Y[i], acc
  node v3  = nop                 role=sink

  edge seq v0 -> v1
  edge seq v1 -> v2
  edge seq v2 -> v3
}

region R1 kind=loop parent=R0 {
  node v4  = nop                 role=source
  region_ref R2
  region_ref R3
  node v5  = lt i, 4             : i1
  node v6  = branch v5                true_child=R2 false_child=R3
  node v7  = nop                 role=sink

  edge seq  v4 -> v5
  edge data v5 -> v6
  edge seq  v6 -> R2 when=true
  edge seq  v6 -> R3 when=false
  edge seq  v6 -> v7
}

region R2 kind=body parent=R1 {
  node v8  = nop                 role=source
  node v9  = mul x, coeff        : i32
  node v10 = add acc, v9         : i32
  node v11 = nop                 role=sink

  edge data v9 -> v10
  edge seq  v8 -> v9
  edge seq  v10 -> v11
}

region R3 kind=empty parent=R1 {
  node v12 = nop                 role=source
  node v13 = nop                 role=sink

  edge seq v12 -> v13
}
----

Future extension:

* For statically timed loops, one possible `gopt` canonicalization is to predicate or inline the `kind=body` region into the surrounding `kind=loop` header SGU when the false alternative is only the trivial `kind=empty` `nop -> nop` region.
* This would turn the loop-header SGU into a fuller one-iteration template while keeping the outer parent `loop` node as the hierarchical summary vertex.
* This transformation is intentionally not part of the current baseline; it remains a possible future loop-dialect simplification.

== Stage: `alloc`

=== Purpose

The `alloc` stage records resource-type allocation decisions before scheduling.

It extends the `seq` structure with:

* one chosen resource type for each operation node
* one concrete initiation interval for each allocated operation node
* one concrete delay for each allocated operation node

The `alloc` stage is the canonical output of allocation and the canonical input to scheduling.
Its output also embeds the normalized executability collateral so later passes can inspect the same FU/op capability data without reopening the original side input.

=== Executability collateral

Allocation consumes one external executability graph supplied by the user.
This collateral may be supplied either as JSON or as one auxiliary `stage exg` µhIR file.

The graph is bipartite:

* one partition contains functional-unit vertices
* the other partition contains canonical µIR operation vertices
* functional-unit vertex names are CAPITALIZED
* operation vertex names are lower case
* each edge states that one operation may execute on one functional-unit vertex
* the weight of an edge carries both `ii` and `d`
* `ii` is the initiation interval of executing that operation on that functional-unit vertex
* `d` is the delay of executing that operation on that functional-unit vertex

Because executability is symmetric, textual and DOT renderings of executability graphs should use undirected edges.

When serialized as µhIR, an executability graph uses:

* `stage exg`
* one or more `region` blocks
* `node <fu_name> = fu partition=fu` for functional-unit vertices
* `node <op_name> = op partition=op` for canonical µIR operation vertices
* undirected `edge exg <fu> -- <op> ii=<int> d=<int>` edges

When `alloc` embeds executability collateral into `alloc.uhir`, it keeps only operation vertices that are relevant to the sequenced design.
It also inserts the synthetic functional unit `CTRL` together with structural operation vertices such as `nop`, `call`, `loop`, `branch`, and `ret` as needed.

A well-formed executability graph must satisfy:

* the functional-unit and operation partitions are disjoint
* every edge connects exactly one functional-unit vertex and one operation vertex
* every canonical µIR operation supported by the language *and* used in the µhIR is connected to at least one functional-unit vertex
* every edge weight defines integer `ii` and integer `d`
* `ii<=d` for all edge weights in the graph

When allocation consumes richer component-library collateral, each executable
operation support should define a *resource type model*.
The resource type model describes the physical operand datatypes expected at the
selected resource boundary and the physical return datatype produced by that
resource.
It should be explicit through a support-local `types` map such as
`operand0=base_t`, `operand1=base_t`, `return=base_t`.
The legacy key `result` may be accepted as an alias for `return`, but new
component libraries should prefer `return` so the support contract clearly
describes what the resource produces and the compiler later consumes.
If `types` is omitted, allocation may infer the same model from the component
port types referenced by the support binding, but explicit support-local types
are preferred because they avoid ambiguous glue behavior.

The operation type carried by `seq` µhIR is the *semantic* operation type.
The resource type model is the *physical* execution contract of the selected
resource.
Allocation and later glue lowering are responsible for provisioning values that
match the support's expected operand datatypes and for consuming the support's
declared return datatype.
If the semantic operation type and physical resource type differ only by legal
integer widening or narrowing, allocation must materialize that adaptation
explicitly in the allocated µhIR graph.
Allocation must not leave this adaptation implicit for µglIR or RTL lowering to
guess later.

Memory supports use the same `types` mechanism, but with the logical memory
object itself kept outside the scalar resource type contract.
The `word_t` parameter is the physical resident word type of the selected
memory resource.
For the current one-element-per-resident-word memory model:

* `load <mem>, <addr> : <semantic-type>` expects `operand1=i32` for the address
  and `return=word_t` for the physical memory word returned by the selected
  memory resource
* `store <mem>, <addr>, <value>` expects `operand1=i32` for the address and
  `operand2=word_t` for the physical memory word written to the selected memory
  resource

The `word_t` parameter is resolved from the selected memory component
specialization, for example `GEN_FF_MEM<word_t=i32,word_len=4>`.
Until sub-word memory lowering exists, `word_t` must match the element type of
the top-level `memref<T, N>`.
If the semantic address, loaded value, or stored value type differs from the
selected memory contract by a legal integer adaptation, `alloc` must materialize
the same `sext`, `zext`, or `trunc` adapter nodes used for arithmetic resources.
Sub-word lane extraction, byte-enable generation, and read-modify-write
adaptation for multiple resident widths are intentionally left to a later
memory model extension.

=== Required elements

A valid `alloc` file must contain:

* `design`
* `stage alloc`
* at least one `region`
* allocated nodes with `class`, `ii`, and `delay`

An `alloc` file may additionally carry one or more embedded `kind=executability` regions.
These regions preserve the executability collateral inside the alloc artifact itself.

=== Required elements in `exg`

A valid `exg` file must contain:

* `design`
* `stage exg`
* at least one `region`
* only `fu` and `op` graph nodes
* only undirected `edge exg ... -- ... ii=<int> d=<int>` edges

=== Required node attributes in `alloc`

Each allocated node should include:

* `class=<resource-type>`
* `ii=<int>`
* `delay=<int>`

Structural conditional-merge nodes such as `sel` may remain after `uhls gopt -p predicate`.
They are treated like control-side logic in the first allocation model and typically receive `class=CTRL ii=0 delay=0`.

Optional attributes:

* width/type information via `: <type>`
* implementation-specific annotations

=== Resource-boundary type adapters

The `alloc` stage is responsible for making resource-boundary type adaptation
explicit.
This is an integrated part of allocation, not a separate optimization pass.

For every resource-backed node, allocation first selects one executable resource
support.
It then compares:

* the semantic µhIR operand/result types of the node
* the expected operand datatypes and declared return datatype of the selected
  resource support

If the types differ by a legal integer adaptation, `alloc` inserts explicit
adapter nodes so the resource-backed operation itself is wired only with values
matching the selected resource type model.

Supported structural adapter opcodes are:

* `sext` — signed integer widening
* `zext` — unsigned integer widening
* `trunc` — integer narrowing by keeping the low bits

The adapter operation is encoded by the node opcode itself.
No separate `adapter_op` attribute is required or allowed by the canonical
form; duplicating the opcode would introduce an unnecessary consistency point.

Adapters are structural datapath nodes.
They should be emitted with:

* `class=ADAPT`
* `ii=0`
* `delay=0`
* `source_type=<type>` — the scalar datatype consumed by the adapter
* `target_type=<type>` — the scalar datatype produced by the adapter

They do not represent user arithmetic and do not consume a normal functional
unit.
Scheduling and binding should preserve their data dependencies while treating
them like zero-cost structural wiring.

Adapter selection is determined by source and destination scalar integer types:

* same width and same signedness: no adapter is needed
* narrower signed source to wider destination: `sext`
* narrower unsigned source to wider destination: `zext`
* wider source to narrower destination: `trunc`
* same width but different signedness: no bit-level adapter is needed, but later
  comparisons or arithmetic must still use the signedness required by their
  selected resource model
* non-integer, aggregate, or memory-reference adaptations are not legal unless a
  later extension defines them explicitly

When an operation result needs narrowing, allocation should keep the
resource-backed operation node as the physical operation and place the semantic
result identity on the result adapter.
This preserves the operation identity while making the value handoff explicit.
In that case the resource-backed node's result type is the selected physical
resource result type, while the adapter's result type is the original semantic
µIR type.

Example:

[source,text]
----
# Before allocation:
node v_mul = mul a_i8, b_i8 : i16

# If allocation selects one physical i32 x i32 -> i32 multiplier:
node v_a32  = sext  a_i8  : i32 class=ADAPT ii=0 delay=0 source_type=i8 target_type=i32
node v_b32  = sext  b_i8  : i32 class=ADAPT ii=0 delay=0 source_type=i8 target_type=i32
node v_mul  = mul   v_a32, v_b32 : i32 class=GEN_MUL_PSEUDOPIPE_II1D3 ii=1 delay=2
node v_i16  = trunc v_mul : i16 class=ADAPT ii=0 delay=0 source_type=i32 target_type=i16
map  v_i16 <- mul_0
----

After adapter materialization, downstream stages may assume:

* every non-adapter resource-backed node's operands match the selected resource
  operand datatypes
* every non-adapter resource-backed node's result type matches the selected
  resource return datatype
* every semantic consumer sees the original µIR type through either the
  resource-backed node itself or an explicit adapter node
* µglIR lowering does not need to invent implicit sign-extension,
  zero-extension, or truncation for resource ports

Allocation must reject a resource support if the required semantic-to-physical
adaptation is not expressible by the adapter rules above.

=== Example: flat allocated form

[source,text]
----
design fir
stage alloc

region R0 kind=procedure {
  node v0 = nop              class=CTRL   ii=0 delay=0 role=source
  node v1 = load A[i]  : i32 class=MEM_RD ii=1 delay=1
  node v2 = mul  v1, c : i32 class=DSP    ii=1 delay=2
  node v3 = ret v2           class=CTRL   ii=0 delay=0
  node v4 = nop              class=CTRL   ii=0 delay=0 role=sink

  edge data v0 -> v1
  edge data v1 -> v2
  edge data v2 -> v3
  edge data v3 -> v4
}
----

== Stage: `sched`

=== Purpose

The `sched` stage records scheduling decisions.

It extends the `alloc` structure with schedule information such as:

* start time
* end time
* overall latency
* optional control guards
* optional schedule style metadata

The `sched` stage is the canonical output of the scheduling pass and the canonical input to binding.

=== Required elements

A valid `sched` file must contain:

* `design`
* `stage sched`
* a schedule declaration
* at least one `region`
* scheduled nodes with `start` and `end`

=== Schedule declaration

Syntax:

[source,text]
----
schedule kind=<control_steps|hierarchical|pipeline>
----

Examples:

[source,text]
----
schedule kind=control_steps
schedule kind=hierarchical
----

=== Required node attributes in `sched`

Each scheduled node should include:

* `class=<resource-class>`
* `delay=<int>`
* `start=<int|timing-expr>`
* `end=<int|timing-expr>`

Timing convention:

* `start` is the first occupied control step
* `end` is the last occupied control step, inclusive
* in fully static schedules, for `delay>0`, `end = start + delay - 1`
* in fully static schedules, for `delay=0`, `end = start`
* for dynamic scheduling extensions, `start`, `end`, `delay`, `steps`, `latency`, and `ii` may carry symbolic timing expressions instead of concrete integers

`timing-expr` is a textual arithmetic sublanguage intended for transfer-format use.
It is restricted to:

* integer literals
* identifiers
* simple function calls such as `max(a, b)`
* unary `-`
* binary `+`, `-`, and `*`
* parentheses

Examples:

* `4`
* `tc`
* `tc * ii`
* `(tc - 1) * ii + rd`

Optional attributes:

* `guard=<predicate>`
* `slot=<int>`
* `stage=<int>` for pipelined forms
* `static_trip_count=<int>` on statically bounded loop hierarchy, typically inferred by `gopt infer_static`
* `iter_latency=<int|timing-expr>`, `iter_initiation_interval=<int|timing-expr>`, and `iter_ramp_down=<int|timing-expr>` on scheduled `loop` nodes when the loop is summarized hierarchically

For hierarchical loop schedules, the outer `loop` node carries the global loop summary.
Its child `kind=loop`, `kind=body`, and `kind=empty` regions remain one-iteration templates with local `start`/`end` times; they are not globally expanded once per trip.
The summarized loop-node delay follows:

[source,text]
----
delay = static_trip_count * iter_initiation_interval + iter_ramp_down
----

For data-independent hierarchical branch scheduling, a `branch` node may summarize its alternatives with:

[source,text]
----
delay = max(delay_true_child, delay_false_child)
----

When a hierarchical control node is fully statically timed, it should carry:

[source,text]
----
timing=static
----

When a hierarchical control node cannot be fully resolved statically at `sched` time, it may carry a symbolic control contract:

[source,text]
----
timing=symbolic
completion=symb_done_<node-id>
----

The node `delay` remains the best available schedule summary. For dynamic `call` and `loop` nodes this is typically:

[source,text]
----
delay=symb_delay_<node-id>
----

Additional control-kind-specific attributes may be attached:

* `branch_condition=<value-id>` on symbolic `branch` nodes
* `continue_condition=<value-id>` on symbolic `loop` nodes
* `iterate_when=<value-id>` on symbolic `loop` nodes when the iterate-again polarity should be explicit
* `exit_when=!<value-id>` on symbolic `loop` nodes when the exit polarity should be explicit
* `handshake=ready_done` on symbolic `call` and `loop` nodes whose datapath side exposes one ordered ready/done contract
* `ready=symb_ready_<node-id>` on symbolic `call` and `loop` nodes

When initiation interval cannot be resolved statically but must still be carried toward later control synthesis, hierarchical `call` and `loop` nodes may also use:

[source,text]
----
ii=symb_ii_<node-id>
----

These attributes are intended as a bridge between `sched` and later `fsm` synthesis. They describe what runtime condition or completion event controls the hierarchy node, without requiring the `sched` stage to fully resolve dynamic control timing.

=== Region schedule summary

Optional but recommended:

[source,text]
----
steps [<start>:<end>]
latency <int|timing-expr>
ii <int|timing-expr>
----

=== Example: flat scheduled form

[source,text]
----
design fir
stage sched
schedule kind=control_steps

region R0 kind=procedure {
  node v1 = load A[i]    : i32 class=memrd delay=1 start=0 end=0
  node v2 = load B[i]    : i32 class=memrd delay=1 start=0 end=0
  node v3 = mul  v1, v2  : i32 class=mul   delay=2 start=1 end=2
  node v4 = add  acc, v3 : i32 class=add   delay=1 start=3 end=3
  node v5 = store C[i], v4    class=memwr delay=1 start=4 end=4

  edge data v1 -> v3
  edge data v2 -> v3
  edge data v3 -> v4
  edge data v4 -> v5

  steps [0:4]
  latency 5
}
----

=== Example: hierarchical scheduled form

[source,text]
----
design branch_example
stage sched
schedule kind=hierarchical

region R0 kind=procedure {
  node c0  = cmp_lt x, y : i1 class=cmp  delay=1 start=0 end=0
  node br0 = branch c0       class=CTRL delay=0 start=0 end=0

  edge control c0  -> br0
  edge seq br0 -> R1 when=true
  edge seq br0 -> R2 when=false
  edge seq R1  -> R3
  edge seq R2  -> R3

  latency 4
}

region R1 kind=then parent=R0 {
  node v1 = add a, b : i32 class=add delay=1 start=1 end=1 guard=true_path
}

region R2 kind=else parent=R0 {
  node v2 = sub a, b : i32 class=sub delay=1 start=1 end=1 guard=false_path
}

region R3 kind=merge parent=R0 {
  node v3 = merge v1, v2 : i32 class=merge delay=0 start=2 end=2
  node v4 = mul v3, 2    : i32 class=mul   delay=2 start=2 end=3

  edge data v3 -> v4
}
----

== Stage: `bind`

=== Purpose

The `bind` stage records implementation decisions after scheduling.

It extends the `sched` form with:

* functional unit bindings
* optional register bindings
* optional memory port bindings
* optional mux declarations

The `bind` stage is the canonical input to `fsm` lowering.

=== Binding modes

Two binding styles are currently expected:

* full operation+register binding
  * concrete scheduled timing
  * functional-unit bindings plus explicit `value ...` register bindings
* FU-only compatibility binding
  * symbolic or dynamic scheduled timing
  * functional-unit bindings, plus optional conservative register bindings for
    mutually exclusive dynamic branch-arm values
  * loop/call-local dynamic register binding remains intentionally conservative

The current built-in binders follow this split:

* `left_edge`
  * interval-based
  * requires concrete integer `start` / `end`
  * emits FU and register bindings
* `compat`
  * compatibility-based
  * accepts symbolic scheduled timing
  * emits FU bindings and may emit conservative branch-exclusive register bindings

Dynamic register binding is only partially defined today. It depends on the
later `fsm` / datapath contract, because symbolic live values may denote either:

* explicitly bound datapath registers, or
* hidden in-flight datapath state behind the ordered ready/done interface

The current conservative notion is one value-storage domain:

* `global`
  * values not nested under a loop/call activation domain
  * these may reuse registers across mutually exclusive dynamic branch arms

Values under `loop` or `call` activation domains are still treated
conservatively and are not dynamically register-bound yet.

So a `bind` artifact with FU bindings but no register bindings is still
well-formed, and a symbolic `compat` artifact may legally contain only the
branch-exclusive subset of register bindings.

Current bind analysis views (`conflict`, `compatibility`, `trp`, `dfgsb`, ...)
still assume concrete scheduled/bound control steps. Symbolic `compat` artifacts
are therefore valid `bind` output, but they are not yet valid input to those
timing-based bind dumps.

=== Required elements

A valid `bind` file must contain:

* `design`
* `stage bind`
* a schedule declaration
* a `resources` block
* at least one `region`
* resource-backed nodes with `bind=<resource-id>`

=== Resources block

Syntax:

[source,text]
----
resources {
  ...
}
----

Supported declarations:

[source,text]
----
fu   <id> : <class-or-component-spec>
reg  <id> : <type>
port <id> : <class-or-component-spec> [<memory-name>]
----

For memory-facing resources, the resource class may carry structural metadata via
one lightweight parameterization form:

[source,text]
----
MEM<word_t=i32,word_len=4>
----

This lets later stages preserve memory shape and word type directly on the
bound resource instead of reconstructing them from separate side metadata.

The same parameterization form may also be used for non-memory components when
allocation has selected one typed instance class.

Example:

[source,text]
----
DIV<base_t=i8>
----

This means:

* the bound resource is still one `DIV`
* but it has been specialized for one concrete `µIR` base type such as `i8`
* later stages may use that semantic parameterization to:
  * resolve port types such as `base_t`
  * select one foreign-HDL parameter override such as `W=8`

Examples:

[source,text]
----
resources {
  fu   mul0  : mul
  fu   add0  : add
  reg  r_acc : i32
  reg  r_tmp : i32
  port mr0   : MEM<word_t=i32,word_len=4> A
  port mw0   : MEM<word_t=i32,word_len=4> C
}
----

=== Required node attributes in `bind`

Each resource-backed operation node should include at least:

* `class=<resource-class>`
* `delay=<int>`
* `start=<int>`
* `end=<int>`
* `bind=<resource-id>`

Structural nodes such as `CTRL` nodes and allocation-materialized `ADAPT`
adapter nodes may remain unbound when they do not consume a concrete
functional-unit resource.

The same inclusive timing convention as `sched` applies:

* `start` is the first occupied control step
* `end` is the last occupied control step, inclusive
* for `delay>0`, `end = start + delay - 1`
* for `delay=0`, `end = start`

=== Value-to-register binding lines

Syntax:

[source,text]
----
value <producer-or-value-id> -> <reg-id> live=[<start>:<end>][,<start>:<end>]*
----

Preferred convention:

* When a region already maps node ids back to earlier IR/source value ids via `map`,
  bind-stage `value ...` lines should use that mapped value id.
* Local node ids remain acceptable as a fallback for hand-written or reduced µhIR
  that does not carry source mappings.

Example:

[source,text]
----
value %t17 -> r_tmp live=[2:3]
----

Well-formedness:

* each interval uses inclusive bounds `[s:e]`
* each interval must satisfy `s<=e`
* one value binding may carry one interval or a comma-separated list of intervals

`value ...` register binding lines are optional in FU-only compatibility-bound
artifacts.

=== Mux declarations

Optional syntax:

[source,text]
----
mux <mux-id> : input=[<src1>, <src2>, ...] output=<dst> sel=<selector>
----

Example:

[source,text]
----
mux mx0 : input=[r_init, r_acc_next] output=r_acc sel=state
----

=== Example: bound design

[source,text]
----
design fir
stage bind
schedule kind=control_steps

resources {
  fu   mul0  : mul
  fu   add0  : add
  port mr0   : memrd A
  port mr1   : memrd B
  port mw0   : memwr C

  reg  r_acc : i32
  reg  r_p   : i32
  reg  r_q   : i32
}

region R0 kind=procedure {
  node v1 = load A[i]    : i32 class=memrd delay=1 start=0 end=0 bind=mr0
  node v2 = load B[i]    : i32 class=memrd delay=1 start=0 end=0 bind=mr1
  node v3 = mul  v1, v2  : i32 class=mul   delay=2 start=1 end=2 bind=mul0
  node v4 = add  acc, v3 : i32 class=add   delay=1 start=3 end=3 bind=add0
  node v5 = store C[i], v4    class=memwr delay=1 start=4 end=4 bind=mw0

  edge data v1 -> v3
  edge data v2 -> v3
  edge data v3 -> v4
  edge data v4 -> v5

  map v1 <- %p
  map v2 <- %q
  map v3 <- %prod
  map v4 <- %acc_next

  value %p        -> r_p   live=[0:1]
  value %q        -> r_q   live=[0:1]
  value %prod     -> r_p   live=[2:3]
  value %acc_next -> r_acc live=[3:4]

  mux mx0 : input=[r_acc, r_next] output=r_acc sel=state
}
----

== Stage: `fsm`

=== Purpose

The `fsm` stage is the first explicit controller-oriented stage after binding.
It is the FSMD stage: the finite-state machine with data specification.

It preserves the bound datapath-oriented µhIR view and adds explicit controller
structure that states:

* the controller encoding
* the external request/response handshake
* the ordered-completion contract
* that overlap is allowed but owned by the datapath

Semantically, this stage is intended to capture:

* the scheduled control flow of the datapath and module interface
* assignment of datapath operations to controller states
* one state-transition step per static scheduled control step
* a valid unrolled state projection for dynamic/hierarchical control
* transitions that may depend on datapath or interface status signals

The currently implemented form already supports explicit controller states,
transitions, and per-state action bundles for the static case, and a first
coarse-grained controller for FU-only symbolic compatibility-bound designs.

=== Required elements

A valid `fsm` file must contain:

* `design`
* `stage fsm`
* a schedule declaration
* a `resources` block
* at least one `controller` block
* at least one `region`

The region-level datapath content follows the same concrete-timing rules as
`bind`.

=== Controller block

Syntax:

[source,text]
----
controller <name> encoding=<binary|one_hot> protocol=req_resp completion_order=in_order overlap=true {
  input  req_valid : i1
  input  resp_ready : i1
  output req_ready : i1
  output resp_valid : i1
  state <state-name> code=<int>
  transition <src> -> <dst> [when=<expr>]
  emit <state-name> [req_ready=true] [resp_valid=true] [issue=[<fu><-<node>, ...>]] [latch=[<reg>, ...]] [select=[<mux><-<sel-signal>, ...>]]
  link <child-controller-name> via=<node-id> [act=[<parent-signal>, <child-signal>]] [ready=[<sched-attr>, <child-signal>]] [done=[<sched-attr>, <child-signal>]] [done_ready=[<parent-signal>, <child-signal>]]
}
----

Current meaning:

* `encoding`
  * chosen controller state encoding
* `protocol=req_resp`
  * external module request/response handshake
* `completion_order=in_order`
  * completions must retire in the same order as accepted inputs
* `overlap=true`
  * overlapping in-flight executions are allowed, but tracked inside the datapath rather than by explicit FSM tags
* `state`
  * one named controller state with one concrete code
* `transition`
  * one state edge, optionally guarded by `when=...`
* `emit`
  * one per-state control/action bundle
  * may include `select=[...]` mux-control outputs when bind-stage `mux ...`
    declarations are available
* `link`
  * one explicit parent/child controller handshake mapping

=== Recursive internal controllers

The current syntax describes both the top-level request/response controllers and
the internal recursive child controllers.

Top-level controllers use:

* `req_valid`
* `resp_ready`
* `req_ready`
* `resp_valid`

Recursive child controllers synthesized for dynamic child hierarchy regions use:

* inputs:
  * `act_valid`
  * `done_ready`
* outputs:
  * `act_ready`
  * `done_valid`

Their intended meaning is:

* admit one child activation on `act_valid && act_ready`
* retire one completed child activation on `done_valid && done_ready`

Parent/child wiring is made explicit through `link ... via=<node-id> ...`
declarations on the parent controller.

=== First Implemented Static Controller Model

The first implemented `fsm` lowering currently synthesizes:

* one `IDLE` state
* one state `Tn` per global concrete control step
* one `DONE` state
* one straight-line transition chain through the control steps
* `emit` actions for:
  * `req_ready` in `IDLE`
  * `resp_valid` in `DONE`
  * FU issue at the start step of each bound operation
  * register latch at the first live step of each bound value binding
  * mux-select control at the latch step of an explicitly declared bound mux

=== First Implemented Dynamic Recursive Hierarchy Model

For symbolic top-level designs with supported dynamic hierarchy nodes, the current `fsm`
lowering now synthesizes:

* one top-level coarse controller over the root region
* one child `protocol=act_done` controller per supported dynamic loop-header,
  callee, or branch-arm controller region set
* one explicit `link` from the parent controller to each synthesized child
  controller

The parent controller still owns coarse progress and post-child issue, while the
child controller owns the local region sequencing. For loops, the child
controller additionally owns the local loop-header/body/exit behavior. For
branches, the child controller owns the true/false arm choice and the chosen
arm-local sequencing.

=== Example: first static controller

[source,text]
----
design add1
stage fsm
schedule kind=control_steps

resources {
  fu ewms0 : EWMS
  reg r_i32_0 : i32
}

controller C0 encoding=binary protocol=req_resp completion_order=in_order overlap=true {
  input  req_valid : i1
  input  resp_ready : i1
  output req_ready : i1
  output resp_valid : i1
  state IDLE code=0
  state T0 code=1
  state T1 code=2
  state T2 code=3
  state DONE code=4
  transition IDLE -> T0 when=req_valid && req_ready
  transition T0 -> T1
  transition T1 -> T2
  transition T2 -> DONE
  transition DONE -> IDLE when=resp_valid && resp_ready
  emit IDLE req_ready=true
  emit T0 issue=[ewms0<-v1]
  emit T1 latch=[r_i32_0]
  emit DONE resp_valid=true
}

region proc_add1 kind=procedure {
  node v0 = nop role=source class=CTRL ii=0 delay=0 start=0 end=0
  node v1 = add x, 1:i32 : i32 class=EWMS ii=1 delay=1 start=0 end=0 bind=ewms0
  node v2 = ret v1 class=CTRL ii=0 delay=0 start=1 end=1
  node v3 = nop role=sink class=CTRL ii=0 delay=0 start=2 end=2

  edge data v0 -> v1
  edge data v1 -> v2
  edge data v2 -> v3

  steps [0:1]
  latency 2
  value v1 -> r_i32_0 live=[1:1]
}
----

== Region-local statements

The following statements may appear inside a `region` block depending on stage.

=== Common

* `node ...`
* `edge ...`
* `map ...`

=== `sched`

Additionally:

* `steps ...`
* `latency ...`
* `ii ...`

=== `bind`

Additionally:

* `value ...`
* `mux ...`

=== `fsm`

Additionally, top-level `controller ...` blocks may appear, while region-local
statements remain the same as `bind`.

== Well-formedness rules

=== Common rules

A µhIR file is well-formed only if:

* exactly one `design` declaration exists
* exactly one `stage` declaration exists
* all referenced regions exist
* all referenced nodes exist
* region identifiers are unique
* node identifiers are unique within the design
* edges reference valid source and destination objects
* stage-specific required attributes are present

=== Stage-specific rules for `seq`

A `seq` file is well-formed only if:

* no `start`, `end`, or `bind` attributes appear unless explicitly allowed by an extension
* no `resources` block appears

=== Stage-specific rules for `exg`

An `exg` file is well-formed only if:

* no ports, constants, schedule block, or resources block appear
* every node has `partition=fu` or `partition=op`
* every `partition=fu` node uses opcode `fu`
* every `partition=op` node uses opcode `op`
* every functional-unit vertex name is CAPITALIZED
* every operation vertex name is lower case
* exg nodes carry no operands or result types
* no `map`, `value`, or `mux` statements appear
* every edge uses kind `exg`
* every edge is undirected
* every edge has integer `ii` and integer `d`

=== Stage-specific rules for `alloc`

An `alloc` file is well-formed only if:

* every allocated node has `class`, `ii`, and `delay`
* every `class=ADAPT` node uses one of `sext`, `zext`, or `trunc`
* every `class=ADAPT` node declares `source_type` and `target_type`
* every `class=ADAPT` node's result type matches its `target_type`
* every resource-backed non-adapter node's operand datatypes and return datatype match the selected resource type model after adapter insertion
* no `start`, `end`, or `bind` attributes appear unless explicitly allowed by an extension
* no `resources` block appears
* any executability graph collateral used to produce the file is well-formed under the rules above
* any embedded `kind=executability` regions are themselves well-formed under the relaxed `alloc`-embedded executability rules above

=== Stage-specific rules for `sched`

A `sched` file is well-formed only if:

* a `schedule` declaration exists
* each scheduled node has `start` and `end`
* `end >= start`
* any region summary such as `steps` or `latency` is consistent with contained nodes

=== Stage-specific rules for `bind`

A `bind` file is well-formed only if:

* a `schedule` declaration exists
* a `resources` block exists
* every bound node has a valid `bind=<resource-id>`
* each referenced bound resource exists in `resources`
* register bindings reference declared registers
* optional mux declarations reference valid sources and outputs

== Provenance and identity preservation

The following should be preserved across the hardware refinement chain:

* region identifiers
* node identifiers
* operation classes
* result types where applicable
* source mapping where available

Recommended invariant:

____
If `v7` denotes a particular multiplication in `seq`, then `v7` should denote the same multiplication in `alloc`, `sched`, and `bind`.
____

== Recommended pass boundary usage

Recommended use of each stage:

* `seq`:
  **input** to allocation

* `alloc`:
  **output** of allocation,
  **input** to scheduling

* `sched`:
  **output** of scheduling,
  **input** to binding

* `bind`:
  **output** of binding,
  **input** to `fsm` lowering

* `fsm`:
  **output** of control/datapath extraction,
  **input** to `glue`

== Suggested implementation strategy

A practical parser strategy is:

* tokenize line by line
* parse headers first
* parse top-level declarations
* parse region and resources blocks
* perform stage-specific validation after parsing

Because syntax is line-oriented and attribute-based, a handwritten Python parser is realistic and appropriate for µhLS.

== Minimal parser-oriented grammar sketch

This section is intentionally informal and implementation-oriented.

[source,text]
----
file        ::= design_decl stage_decl top_decl*
top_decl    ::= global_decl | schedule_decl | resources_block | controller_decl | region_decl

design_decl ::= "design" IDENT
stage_decl  ::= "stage" ("seq" | "alloc" | "sched" | "bind" | "fsm")

global_decl ::= ("input" | "output") IDENT ":" TYPE
              | "const" IDENT "=" LITERAL ":" TYPE

schedule_decl ::= "schedule" "kind=" IDENT

resources_block ::= "resources" "{" resource_decl* "}"
resource_decl   ::= "fu" IDENT ":" IDENT
                  | "reg" IDENT ":" TYPE
                  | "port" IDENT ":" IDENT [IDENT]

controller_decl ::= "controller" IDENT attr_list? "{" controller_stmt* "}"
controller_stmt ::= ("input" | "output") IDENT ":" TYPE
                  | "state" IDENT attr_list?
                  | "transition" IDENT "->" IDENT attr_list?
                  | "emit" IDENT attr_list?

region_decl ::= "region" IDENT attr_list? "{" region_stmt* "}"

region_stmt ::= node_decl
              | edge_decl
              | map_decl
              | steps_decl
              | latency_decl
              | ii_decl
              | value_decl
              | mux_decl
              | region_ref_decl

node_decl  ::= "node" IDENT "=" OP ARGS [ ":" TYPE ] attr_list
edge_decl  ::= "edge" IDENT REF ("->" | "--") REF [ "when=" IDENT ]
map_decl   ::= "map" IDENT "<-" REF
steps_decl ::= "steps" "[" INT ":" INT "]"
steps_decl ::= "steps" "[" timing_expr ":" timing_expr "]"
latency_decl ::= "latency" timing_expr
ii_decl    ::= "ii" timing_expr
value_decl ::= "value" REF "->" IDENT "live=" live_interval { "," live_interval }
live_interval ::= "[" INT ":" INT "]"
mux_decl   ::= "mux" IDENT ":" "input=" "[" list "]" "output=" IDENT "sel=" IDENT
region_ref_decl ::= "region_ref" IDENT

timing_expr ::= timing_sum
timing_sum ::= timing_product { ("+" | "-") timing_product }
timing_product ::= timing_factor { "*" timing_factor }
timing_factor ::= INT
                | IDENT
                | IDENT "(" timing_expr { "," timing_expr } ")"
                | "-" timing_factor
                | "(" timing_expr ")"
----

== Summary

µhIR provides a unified textual representation family for hardware-side refinement in µhLS.

The common `.uhir` extension is paired with an internal stage declaration:

* `stage seq`
* `stage alloc`
* `stage sched`
* `stage bind`

This design keeps the file family:

* human-readable
* parser-friendly
* stable across passes
* suitable for hierarchical scheduling and downstream binding
