= µhLS µC Source Language Specification
:doctype: article
:toc:
:toclevels: 3
:sectnums:

µC is the restricted source language of µhLS.
It is designed to be small enough for students to understand completely, while still expressive enough to generate interesting control-flow, dataflow, optimization, and scheduling problems.

µC is intentionally *not* a full C dialect.
It is a structured, side-effect-limited subset that lowers deterministically into canonical µIR.

== Design goals

µC is designed to support:

* scalar integer computation
* structured control flow
* static arrays and explicit memory access
* deterministic lowering into block-based scalar-SSA µIR
* compiler middle-end analyses and optimizations
* high-level synthesis of simple computational kernels

µC is intentionally restricted so that students can focus on:

* IR construction
* CFG construction
* dataflow analysis
* optimization
* dependence analysis
* scheduling and binding
* software code generation

rather than on the complexity of full C.

== Program model

A µC program consists of one or more function definitions.

In the base µC subset:

* one function is typically treated as the synthesis entry function
* user-defined function calls are not allowed
* recursion is not allowed
* dynamic dispatch is not allowed

This keeps frontend lowering and HLS behavior simple.

== Types

µC supports only a small set of scalar integer and array types.

Supported scalar types:

* `bool`
* `int8_t`, `int16_t`, `int32_t`
* `uint8_t`, `uint16_t`, `uint32_t`

Supported aggregate-like types:

* fixed-size local arrays, for example `int32_t A[16];`
* array parameters, for example `int32_t A[16]`

Not supported:

* floating-point types
* structs
* unions
* enums
* pointers
* function pointers
* typedef-heavy abstractions beyond the fixed integer types above

== Objects and storage

µC supports the following kinds of objects:

* scalar function parameters
* array function parameters
* scalar local variables
* fixed-size local arrays

Restrictions:

* no global variables in the base language
* no static locals
* no dynamic allocation
* no heap
* no pointer-based memory access

These restrictions ensure that memory can be lowered cleanly into explicit array `load` and `store` operations in µIR.

== Expressions

µC expressions are side-effect free.

This means that expressions may compute values, but they may not:

* assign to variables
* modify arrays
* perform function calls
* contain hidden sequencing effects

=== Allowed expression forms

Allowed expression forms include:

* typed integer literals
* boolean literals
* scalar variable references
* array element reads, for example `A[i]`
* parenthesized expressions
* unary operations
* binary operations
* comparisons

=== Allowed unary operators

* unary minus: `-x`
* bitwise not: `~x`
* logical not: `!x`

=== Allowed binary arithmetic operators

* `+`
* `-`
* `*`
* `/`
* `%`

=== Allowed bitwise and shift operators

* `&`
* `|`
* `^`
* `<<`
* `>>`

=== Allowed comparison operators

* `<`
* `<=`
* `>`
* `>=`
* `==`
* `!=`

== Statements

µC supports only structured statements.

=== Allowed statements

* declaration with optional initialization
* scalar assignment
* array element assignment
* `if` / `else`
* canonical `for` loop
* `return`
* nested compound blocks

Examples:

[source,c]
----
int32_t x = 0;
uint32_t y;
A[i] = x + y;
if (x < y) { ... } else { ... }
for (i = 0; i < N; i = i + 1) { ... }
return x;
----

=== Not supported

* `while`
* `do while`
* `switch`
* `goto`
* labels
* `break`
* `continue`
* `++` / `--`
* comma operator
* ternary `?:` operator
* inline assembly

These constructs are excluded to keep control-flow lowering simple and canonical.

== Function definitions

A µC function has:

* a name
* typed parameters
* one scalar return type
* a compound statement body

Restrictions:

* no variadic functions
* no recursion
* no multiple return values
* no user-defined calls in the base language

Example:

[source,c]
----
int32_t dot4(int32_t A[4], int32_t B[4]) {
    int32_t i;
    int32_t sum = 0;

    for (i = 0; i < 4; i = i + 1) {
        sum = sum + A[i] * B[i];
    }

    return sum;
}
----

== Canonical loop form

The only loop form supported in base µC is the canonical `for` loop.

Required shape:

[source,c]
----
for (i = init; i < bound; i = i + step) {
    ...
}
----

or equivalent forms using `<=`, `>`, or `>=` where explicitly allowed by the frontend.

Restrictions for the frontend:

* the induction variable is a scalar integer variable
* initialization is a simple assignment
* step is affine and explicit
* loop condition is a simple comparison
* no `break` or `continue`
* no hidden side effects in the header

This ensures loops lower cleanly into explicit control-flow graphs and are suitable for dependence and scheduling analyses.

== Array and memory model

Arrays are the only mutable memory objects in µC besides scalar variables.

Rules:

* all arrays have statically known element type
* local arrays have statically known size
* array accesses use explicit indices
* indices are scalar integer expressions
* distinct arrays are assumed not to alias

Recommended pedagogical restriction:

* array index expressions should be affine or simple arithmetic forms such as `i`, `i+1`, or `2*i+3`

This restriction is especially helpful for later dependence analysis and HLS scheduling.

== Control-flow lowering intent

µC is designed to lower into µIR with the following properties:

* all structured control flow becomes explicit basic blocks
* all control-flow edges are explicit
* no implicit fallthrough exists after lowering
* every µIR block ends with an explicit terminator

This is a defining property of the µhLS pipeline.

== Deterministic lowering expectations

A µC frontend should lower programs into µIR in a deterministic and canonical way.

In particular:

* complex expressions are decomposed into three-address steps
* all array accesses become explicit `load` or `store`
* all conditionals become explicit branch blocks
* all loops become explicit headers, bodies, latches, and exits as needed
* no hidden control transfers remain

This keeps later passes independent of frontend syntax details.

== Undefined and implementation-defined behavior

The minimalistic frontend should avoid inheriting the full complexity of ISO C.

Recommended rule:

* µC has simple, explicit reduced semantics rather than full C semantics

In particular, the implementation should define clearly:

* integer widths
* signedness of each type
* shift behavior
* boolean representation
* array bounds expectations

A simple and useful course policy is:

* well-formed test programs are assumed not to overflow in unintended ways
* out-of-bounds array access is invalid
* unsupported constructs are rejected by the frontend

== Forbidden constructs summary

The following are not part of base µC:

* user-defined function calls
* recursion
* pointers
* pointer arithmetic
* dynamic memory allocation
* structs, unions, and floating point
* `while`, `do while`, `switch`, `goto`
* `break`, `continue`
* side-effecting expressions
* function pointers
* indirect memory access through pointers

== Intended relation to µIR

µC is the source language.

It lowers into:

* *µIR*, the canonical typed, block-based scalar-SSA IR with explicit control-flow edges and explicit memory operations

The intended pipeline is:

[source,text]
----
µC
  -> AST
  -> µIR
  -> CFG canonicalization
  -> optimization / analysis / scheduling
----

== Integrated testbenches in µIR

To support integrated testbenches, one translation unit may contain both:

* one or more callable kernel functions such as `dot4`
* a `main(void) -> i32` testbench entry function

The current restrictions are:

* `main` returns `int32_t`
* `main` takes no arguments
* the only supported pseudo-standard-library side effect is `uhls_printf(...)` for debugging with the interpreter or later lowering to SystemVerilog print statements.

When lowered to µIR, this introduces two extra surface forms:

* `local <name>[<size>]:<elemtype>`
  Declares one function-local array object available to `load`, `store`, and array-valued call operands.
* `print "<format>"[, <value0>, <value1>, ...]`
  Represents `uhls_printf` after lowering and consumes scalar SSA operands.

Example:

[source,text]
----
func main() -> i32

local main$A[4]:i32
local main$B[4]:i32

block entry:
    store main$A[0:i32], 1:i32
    store main$B[0:i32], 4:i32
    res_0:i32 = call dot4(main$A, main$B)
    print "Unexpected return value: %d", res_0
    ret 1:i32
----

Verification expectations:

* every `local` declaration is scoped to its containing function
* local array names are valid array symbols anywhere inside that function
* `print` operands must already be valid scalar SSA values
* array-valued call operands may reference parameters or `local` arrays

== Example µC kernels

The following kernels are appropriate for µC:

* dot product
* FIR filter kernels
* SAXPY-style vector kernels
* reductions such as min, max, or sum
* population count variants
* fixed-size convolution fragments
* small AES round fragments
* simple matrix micro-kernels

These are rich enough for optimization and scheduling, while still fitting the language restrictions.
