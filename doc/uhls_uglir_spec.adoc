= µhLS µglIR hardware glue IR specification
:toc:
:toclevels: 3
:sectnums:

.tldr:
[quote]
____
µglIR is the structural, signal-oriented IR that µhLS emits after `fsm.uhir` and before `rtl` lowering. It captures the hardware glue of a design, such as registers, nets, muxes, controller state, component instances, and their explicit connectivity, while leaving arithmetic and memory internals inside referenced components.
____

== Overview

This document specifies the *µglIR* textual intermediate representation used by
*µhLS* after `fsm.uhir` lowering and before `rtl` emission.

The common file extension is:

[source,text]
----
.uglir
----

Typical filenames are:

[source,text]
----
fir.uglir
dot4.uglir
----

The intended backend flow becomes:

[source,text]
----
µIR
  -> seq.uhir
  -> alloc.uhir
  -> sched.uhir
  -> bind.uhir
  -> fsm.uhir
  -> .uglir
  -> rtl
  -> Verilog
  -> Yosys / ASIC flow
----

Its role is to describe the *hardware glue structure* of one design:

* registers
* nets
* muxes
* component instances
* controller state storage
* explicit signal connectivity

It does *not* restate the internals of arithmetic or memory components.

== Purpose

µglIR is the first backend IR within µhLS whose primary viewpoint is
*signal-oriented hardware structure* instead of graph refinement or control-step
actions.

It exists to separate:

* FSMD semantics
  * states
  * transitions
  * actions such as `issue`, `latch`, `select`

from:

* RTL syntax
  * Verilog module syntax
  * always-block shape
  * blocking/non-blocking conventions
  * target-specific code style

µglIR describes how µhLS components are glued together, not how those
components compute internally. An adder, multiplier, memory port, or child
controller therefore appears as an *instance with ports*, not as re-expanded
behavioral logic.

== Current Memory Assumption

For the first µglIR version, each top-level addressed memory object is lowered
as one visible memory-interface bundle, not as storage instantiated inside the
bare core.

So a top-level `memref<T>` or `memref<T, N>` such as `A` may expand into
glue-level ports like:

* `A_addr`
* `A_wdata`
* `A_we`
* `A_rdata`

If the original top-level memory had one known extent `N`, µglIR should keep the
raw bundle shape above but preserve the depth as explicit metadata, for example
via parameterized memory-interface resources such as:

[source,text]
----
port A : MEM<word_t=i32,word_len=4> A
----

Generated constants such as `A_depth` may still exist as convenient lowering
artifacts, but the structured memory metadata should survive primarily on the
resource itself.
That metadata can then be consumed by RTL wrapper generation to size wrapper-local
memories and address windows without changing the bare core interface shape.

The shared component library may declare such parameters explicitly, for example:

[source,json]
----
"MEM": {
  "kind": "memory",
  "parameters": {
    "word_t": { "kind": "type" },
    "word_len": { "kind": "int" }
  }
}
----

Lowering should validate concrete resource specs such as
`MEM<word_t=i32,word_len=4>` against that declaration instead of treating the
parameter list as an opaque string.

The same mechanism also applies to typed arithmetic components.

Example:

[source,json]
----
"DIV": {
  "parameters": {
    "base_t": { "kind": "type", "required": true }
  }
}
----

One concrete bound resource may then carry a semantic specialization such as:

[source,text]
----
DIV<base_t=i8>
----

This is intentionally written in terms of `µIR`-level types such as `i8`,
`i16`, or `i32`, not in terms of foreign-HDL parameter names such as `W`.

The shared component library may also carry foreign HDL linkage metadata for
components that are implemented externally and should not be lowered into
`µglIR` internals. A typical imported entry may include:

[source,json]
----
"ALU": {
  "hdl": {
    "language": "verilog",
    "module": "ALU",
    "source": "rtl/alu.v"
  }
}
----

This linkage metadata identifies:

* the HDL family
* the external module/entity name
* the source file path recorded in the library collateral

`µglIR` should use that information only as component linkage metadata. It
should not import or rewrite the foreign HDL implementation itself.

For parameterized foreign modules, the library may additionally define an HDL
parameter mapping from semantic component parameters to concrete HDL instance
parameters.

Example:

[source,json]
----
"DIV": {
  "hdl": {
    "language": "verilog",
    "module": "DIV",
    "source": "test_div.v",
    "parameters": {
      "W": "$bits(base_t)"
    }
  }
}
----

In that example:

* `base_t` is the semantic component parameter used by `µhLS`
* `W` is the Verilog module parameter used by the foreign HDL
* `$bits(base_t)` maps one semantic type such as `i8` to one concrete integer
  width such as `8`

So one semantic resource choice like:

[source,text]
----
DIV<base_t=i8>
----

may become one concrete HDL instance like:

[source,text]
----
DIV<W=8>
----

The component library should also declare which typed `µIR` operations one
parameterized component instance may execute.
Each support entry's `types` map is the resource contract:

* `operandN` entries are the datatypes the resource expects at its input
  boundary
* `return` is the datatype the resource produces for the compiler to consume
* the legacy key `result` may be accepted as an alias for `return`, but new
  component libraries should prefer `return`

Allocation and later glue lowering are responsible for provisioning operands
that match the declared operand datatypes and for consuming the declared return
datatype.
If a semantic µIR operation has narrower or wider integer types than the
selected support declares, `alloc` must materialize explicit µhIR
resource-boundary adapters before glue lowering.
These adapters are `class=ADAPT` µhIR nodes whose opcode is one of `sext`,
`zext`, or `trunc`; their `source_type` and `target_type` attributes record
the typed handoff being materialized.

Example:

[source,json]
----
"supports": {
  "div": {
    "ii": 1,
    "d": 3,
    "types": {
      "operand0": "base_t",
      "operand1": "base_t",
      "return": "base_t"
    }
  }
}
----

This means:

* `DIV<base_t=i8>` matches `div(i8, i8) -> i8`
* `DIV<base_t=i32>` matches `div(i32, i32) -> i32`
* `alloc` must not place one mismatched `i32` operation onto one `DIV<base_t=i8>`
  instance unless the µhIR `alloc` stage has materialized explicit
  resource-boundary adapter nodes that satisfy the selected resource type model

When the shared component library classifies components by `kind`, that kind
should reflect the execution contract rather than only the implementation
language:

* `combinational`
  * intended for non-handshaked components without internal state/registers
  * outputs are boundary-visible combinational ports
* `sequential`
  * intended for non-overlapped multi-cycle components with internal state
  * concrete supports should therefore satisfy `ii >= d`
* `pipelined`
  * intended for overlapped multi-cycle components with internal pipeline state
  * concrete supports may use `ii < d`
* `memory`
  * addressed memory-interface component

The timing numbers on one support entry should be interpreted at the *module
boundary*:

* `d`
  * output-port visibility delay
  * counted in control cycles from the issue cycle up to the cycle whose end may
    latch the result from the FU output port
* `ii`
  * minimum issue spacing at the same component boundary

Registered outputs are therefore *not* implied by `kind=pipelined` or
`kind=sequential`.

A multicycle combinational FU and a multicycle pipelined FU may both legally use
the same `d` when they make the result visible at their output ports in the same
cycle. The difference is whether the FU uses internal state/registers to realize
that delay.

For one multicycle `combinational` FU, the surrounding glue must therefore keep
its inputs stable across the control-step window covered by `d`.

In other words:

* the FU itself does not capture operands internally
* so the generated glue must hold the operand values until the output-port delay
  has elapsed and the scheduled consumer may latch the result

If one foreign HDL component exposes *registered outputs*, that extra cycle must
already be reflected in its declared `d`.

When one produced value is bound to one storage register, µglIR lowering should
distinguish two related but different moments:

* `capture_cycle`
  * the controller cycle whose end latches the FU output into the bound register
* `live_start`
  * the first later cycle in which downstream logic may consume that bound
    register value

These are often equal for directly consumable combinational sources, but they do
not have to be equal.

In particular:

* for `pipelined` and `sequential` FUs, the glue may need to capture the FU
  result one cycle before the first consumer-visible `live_start`
* later consumers should then read the bound register in `live_start`, not
  bypass the register back to the FU output

This is the rule that lets one value be:

* captured at the end of `T2`
* then first consumed from its bound register in `T3`

Example:

* `d=2`
  * `ccx+0`
    * issue
    * operands are already stable during the cycle
  * `ccx+1`
    * final internal step completes
    * result is visible on the output port
    * a consumer register may latch it at the end of `ccx+1`
  * `ccx+2`
    * later logic may consume the bound register value if the design inserted
      storage between FUs

So, for example, a divider with `ii=1` and `d=3` may still be classified as
`combinational` if that delay is realized without internal state. It should be
classified as `pipelined` only when overlapping operation or internal pipeline
state is part of the contract.

== Typed Component Parameterization

The preferred style for parameterized functional units is:

* use semantic component parameters such as `base_t`
* express support compatibility in terms of expected `µIR` operand datatypes and
  declared return datatypes
* map semantic parameters to foreign-HDL parameters only at the HDL linkage
  boundary

This keeps:

* `alloc`
  * aware of typed compatibility
* `µglIR`
  * aware of concrete semantic specialization such as `DIV<base_t=i8>`
* `rtl`
  * responsible for translating that specialization to the concrete HDL module
    parameter override such as `DIV #(.W(8)) ...`

This split avoids making front-end and allocation logic depend directly on
Verilog-specific parameter names.

Current assumption:

* one visible memory-interface bundle per top-level memory name
* multiple scheduled/bound accesses may share that bundle when they are
  multiplexed over time onto the same bound memory resource
* competing accesses to the same memory bundle in one FSMD state are currently
  invalid and should be rejected before or during µglIR lowering
* explicit multi-port or arbitrated memory modeling is not yet part of µglIR

This keeps the first memory lowering simple while still allowing repeated
time-separated accesses to one memory through one port.

In particular, the memory-bound FU/resource used during scheduling and binding
should be understood as an access endpoint, not necessarily as a concrete
storage instance in the emitted bare-core RTL.

== Design Goals

µglIR is designed to be:

* human-readable
* line-oriented
* easy to diff
* explicit about connectivity
* explicit about sequential state
* explicit about controller/datapath interfacing
* easy to lower into Verilog mechanically

It should also serve as the main validation point for hardware structure before
RTL printing.

== Address Maps

Wrapped `µglIR` designs may carry explicit software-visible address maps.

The first use is wrapper/protocol composition, where `glue` generates an
address map from the protocol plan so the wrapper register and memory layout is
visible in the exchange artifact itself instead of being only implicit in the
RTL emitter.

Example:

[source,text]
----
address_map wishbone {
  register control_status offset=32'h0000_0000 access=rw symbol=WB_REG_CONTROL_STATUS
  register x offset=32'h0000_0100 access=rw symbol=WB_REG_IN_X type=i32
  register result offset=32'h0000_0200 access=ro symbol=WB_REG_OUT_RESULT type=i32
  memory A offset=32'h0000_1000 span=32'h0000_0010 access=rw symbol=WB_MEM_A_BASE word_t=i32 depth=4
}
----

Intended meaning:

* `register`
  * one scalar software-visible register location
* `memory`
  * one software-visible memory window
* `offset`
  * protocol-relative base offset
* `span`
  * byte span for one memory window
* `access`
  * software-visible access policy such as `rw`, `ro`, `rw_err`, `ro_err`
* `symbol`
  * corresponding generated symbolic constant name

The current `rtl` backends do not infer behavior from the `address_map` block
alone; it is generated metadata for software/integration tooling and for
keeping wrapper layout explicit in the `µglIR` exchange artifact.

== Generated Signal Naming Convention

The generated µglIR emitted by µhLS uses a simple storage-versus-combinational
naming convention:

* register resources/signals end in `_q`
* net resources/signals end in `_n`
* mux resources/signals end in `_n`

Examples:

* `state_q`
* `next_state_n`
* `r_acc_q`
* `mx_r_acc_n`
* `alu0_a_n`
* `const_i32_0_n`

This convention is intended to make the hardware role of one signal visible at
a glance:

* `_q` means stored state
* `_n` means next/combinational/interconnect logic

Top-level interface ports and instance identifiers remain unsuffixed.

== Position Relative To `fsm`

`fsm.uhir` is *action-oriented*.

It says:

* which states exist
* when transitions fire
* which actions happen in each state

µglIR is *signal-oriented*.

It says:

* which registers, nets, muxes, and instances exist
* which signals drive which ports
* which enables and select lines are asserted
* which state register implements the controller
* how sequential updates are applied

In short:

* `fsm` describes *control intent*
* `uglir` describes *hardware realization*

== Common Structure

All µglIR files share one common outer structure.

A file contains:

* one `design` declaration
* one `stage uglir` declaration
* optional port declarations
* optional constant declarations
* one `resources` block
* zero or more structural declaration blocks
* one or more sequential/combinational logic blocks

Minimal example:

[source,text]
----
design add1
stage uglir

input  clk : clock
input  rst : i1
input  req_valid : i1
output req_ready : i1

resources {
  reg state : u2 reset=0
  reg r_acc : i32
  net next_state : u2
  inst ewms0 : EWMS
}
----

== Common Lexical Conventions

=== Comments

Comments begin with `#` and continue to the end of the line.

[source,text]
----
# this is a comment
----

=== Identifiers

Identifiers are used for:

* design names
* ports
* registers
* nets
* muxes
* instances
* controller states
* symbolic control signals

Examples:

[source,text]
----
dot4
req_valid
state
r_acc
mx0
ewms0
DONE
done_valid
----

Recommended style:

* registers: `r_acc_q`, `state_q`
* nets: `next_state_n`, `req_fire_n`
* muxes: `mx0_n`, `mx_r_acc_n`
* instances: `ewms0`, `memA`

=== Types

Scalar examples:

[source,text]
----
i1
i8
i16
i32
u2
u3
u8
u16
u32
clock
----

Aggregate and interface-like types may be introduced later, but the first
version should prefer explicit scalar ports and explicit component instances.

=== Attribute Syntax

Attributes use `key=value`.

Examples:

[source,text]
----
reset=0
width=2
kind=state
----

Lists use brackets:

[source,text]
----
input=[r_acc, ewms0_y]
----

== Top-Level Declarations

=== Design Declaration

Every file must begin with exactly one design declaration.

Syntax:

[source,text]
----
design <design-name>
----

=== Stage Declaration

Every file must declare exactly one stage.

Syntax:

[source,text]
----
stage uglir
----

=== Port Declarations

Ports describe the hardware module interface.

Syntax:

[source,text]
----
input  <name> : <type>
output <name> : <type>
----

Recommended top-level handshake names are:

* `req_valid`
* `req_ready`
* `resp_valid`
* `resp_ready`

Clock/reset are explicit ports, typically:

* `clk : clock`
* `rst : i1`

=== Constants

Constant declarations may appear at top-level when one named literal is useful
across multiple assignments.

Syntax:

[source,text]
----
const <name> = <value> : <type>
----

Example:

[source,text]
----
const IDLE = 0 : u3
----

== Resources Block

The `resources` block declares named hardware objects.

Syntax:

[source,text]
----
resources {
  reg  <name> : <type> [reset=<literal>]
  net  <name> : <type>
  inst <name> : <component-kind-or-spec>
  port <name> : <component-kind-or-spec> [<external-name>]
  mux  <name> : <type>
}
----

Example:

[source,text]
----
resources {
  reg  state : u3 reset=0
  reg  r_i32_0 : i32
  net  next_state : u3
  net  req_fire : i1
  inst ewms0 : EWMS
  port A : MEM<word_t=i32,word_len=4> A
  mux  mx_r_i32_0 : i32
}
----

Semantics:

* `reg`
  * clocked storage element
* `net`
  * combinational signal
* `inst`
  * external datapath or controller component instance
* `port`
  * external protocol or memory-facing interface resource
  * may carry structural parameters such as `word_t` and `word_len`
* `mux`
  * explicit multiplexer object, later lowered to combinational selection logic

== Structural Statements

The first version of µglIR should support four core structural statements:

* `assign`
* instance port attachment
* `mux`
* `link`

=== Instance Port Attachment

Instance wiring use one explicit application-like form:

Syntax:

[source,text]
----
<inst>.<port>(<net>)
----

Examples:

[source,text]
----
ewms0.a(r_i32_0)
ewms0.b(x_in)
ewms0.y(ewms0_y)
----

Rules:

* the argument must be one named `net`, `reg`, or top-level port name
* direct port-to-port hookups are not allowed
* forms such as `inst0.out(inst1.in)` are illegal
* when one instance output must feed another instance input, µglIR must first
  declare one helper `net` and attach both ports to that net

Example:

[source,text]
----
net ewms0_y : i32
ewms0.y(ewms0_y)
mx_r_acc.in1(ewms0_y)
----

=== Assign

`assign` defines one combinational named signal.

Syntax:

[source,text]
----
assign <name> = <expr>
----

Long expressions may be wrapped across multiple physical lines by enclosing the
expression in one balanced parenthesized form. The parser then treats the
multiline form as one logical expression.

Example:

[source,text]
----
assign next_state =
  (
    state == IDLE ? (req_fire ? T0 : IDLE) :
    state == T0   ? T1 :
                    DONE
  )
----

Examples:

[source,text]
----
assign req_fire = req_valid & req_ready
assign resp_fire = resp_valid & resp_ready
assign req_ready = (state == IDLE)
assign result = r_i32_1
----

The expression language should initially remain small:

* literals
* signal references
* unary `!`
* binary `&`, `|`, `^`
* equality/inequality
* simple concatenation or slices later if needed

=== Mux

`mux` declares the explicit source alternatives for one selected datapath signal.

Syntax:

[source,text]
----
mux <name> : <type> sel=<select-signal> {
  <key> -> <source>
  <key> -> <source>
  ...
}
----

Example:

[source,text]
----
mux mx_r_acc : i32 sel=sel_r_acc {
  hold  -> r_acc
  ewms0 -> ewms0.y
}
----

Semantics:

* the mux is explicit structural glue
* the select signal is named separately
* downstream instance port attachments or sequential updates consume the mux output

=== Link

`link` expresses one explicit controller/datapath or parent/child-controller
glue relation.

Syntax:

[source,text]
----
link <instance-or-controller> {
  <lhs> <-> <rhs>
  ...
}
----

Example:

[source,text]
----
link C_loop_hdr {
  act_valid  <-> activate_v4
  act_ready  <-> ready_v4
  done_valid <-> done_v4
  done_ready <-> resp_ready
}
----

The exact textual form may evolve, but µglIR should make such handshake wiring
explicit instead of hiding it in naming conventions.

== Sequential Logic

µglIR should make state updates explicit.

Syntax:

[source,text]
----
seq <clock-name> {
  if <reset-signal> {
    <reg> <= <value>
    ...
  } else {
    <reg> <= <value>
    if <enable> {
      <reg> <= <value>
    }
  }
}
----

As with `assign`, long reset conditions, enables, and update expressions may be
wrapped across multiple physical lines inside balanced parentheses.

Example:

[source,text]
----
seq clk {
  if rst {
    state <= IDLE
    r_i32_0 <= 0:i32
  } else {
    state <= next_state
    if latch_r_i32_0 {
      r_i32_0 <= mx_r_i32_0
    }
  }
}
----

Semantics:

* all `reg` updates occur synchronously on the named clock
* explicit enables model FSMD `latch=[...]`
* the controller state register is just another register in µglIR

== Combinational Controller Logic

Controller next-state and output decode logic should appear as explicit
combinational assignments.

Example:

[source,text]
----
assign req_ready = (state == IDLE)
assign resp_valid = (state == DONE)

assign next_state =
  state == IDLE ? (req_fire ? T0 : IDLE) :
  state == T0   ? T1 :
  state == T1   ? DONE :
                  IDLE
----

The exact expression syntax may later be normalized with explicit `switch`
blocks, but the semantics should remain combinational and explicit.

== Controller/Datapath Contract

µglIR assumes the following split:

* controller state and transition logic are explicit in µglIR
* datapath component internals are opaque instance behavior
* muxes, register enables, and handshake glue are explicit in µglIR

So one FSMD action bundle such as:

[source,text]
----
emit T2 issue=[ewms0<-v2] latch=[r_acc] select=[mx_r_acc<-ewms0]
----

would lower conceptually to:

* one or more instance port attachments driving `ewms0`
* one mux select assignment for `mx_r_acc`
* one latch-enable signal for `r_acc`
* one sequential guarded update of `r_acc`

== Relationship To Components

µglIR does not redefine component functionality.

For example:

* `inst ewms0 : EWMS`

means:

* one instance of component kind `EWMS` exists
* µglIR must attach its inputs/outputs to named nets
* µglIR need not restate how `EWMS` computes its result

This keeps µglIR focused on *glue logic*.

== Initial Lowering Expectations

The first `fsm -> uglir` lowering should be expected to produce:

* top-level ports
* one state register and next-state net per controller
* request/response handshake nets
* one register object per bound register
* one instance per bound FU or memory port
* one explicit mux/select object where routing is needed
* explicit sequential updates for controller state and data registers
* explicit parent/child controller links for recursive dynamic control

== Example

[source,text]
----
design add1
stage uglir

input  clk : clock
input  rst : i1
input  req_valid : i1
input  resp_ready : i1
input  x : i32
output req_ready : i1
output resp_valid : i1
output result : i32

resources {
  reg  state : u3 reset=0
  reg  r_i32_0 : i32
  net  next_state : u3
  net  req_fire : i1
  net  resp_fire : i1
  net  latch_r_i32_0 : i1
  net  ewms0_y : i32
  inst ewms0 : EWMS
}

assign req_fire = req_valid & req_ready
assign resp_fire = resp_valid & resp_ready
assign req_ready = (state == IDLE)
assign resp_valid = (state == DONE)

ewms0.a(x)
ewms0.y(ewms0_y)
assign result = r_i32_0

seq clk {
  if rst {
    state <= IDLE
    r_i32_0 <= 0:i32
  } else {
    state <= next_state
    if latch_r_i32_0 {
      r_i32_0 <= ewms0_y
    }
  }
}
----

== Open Items

The following areas are intentionally left open for later refinement:

* exact expression grammar for combinational logic
* whether muxes are always explicit objects or may be inlined into `assign`
* concrete syntax for controller links
* memory-port instance conventions
* whether top-level output driving should always use `assign`
* reset polarity/style conventions
* multi-clock designs
* hierarchy flattening versus nested module-style µglIR

These should be resolved only when the first `fsm -> uglir` lowering is
implemented.
